## 2. Стандарты именования

### 2.1. Файлы
- **Стиль**: `Название-Файла.md` (Title-Case с дефисами).
- **Запрещено**: `ALL-CAPS`, `нижние_подчеркивания`, `пробелы`, цифровые префиксы в именах файлов.
- **Язык**: Русский (для смысловых названий), Английский (для терминов: `RFM-Анализ`, `API-Ключи`, `PnL-Отчёт`).

### 2.2. Папки
- **Корневые**: `###-Название-Папки` (трёхзначный код).
- **Подпапки модулей**: `##-Название-Отдела` (двузначный код).

## 4. Жизненный цикл документа

1. Новый документ создается **обязательно** из соответствующего шаблона (`070-Шаблоны`).
    - **Tpl-Модуль**: для обычных фич.
    - **Tpl-Тех-Паспорт**: для критических систем (платежи, ядро).
    - **Tpl-Решение**: для фиксации архитектурного выбора (ADR).
    - **Tpl-Сессия**: для журналов работы с ИИ.
    - **Tpl-UI-Компонент**: для описания элементов дизайн-системы.
2. При создании фичи файл переводится из статуса `draft` в `plan`.
3. После реализации и тестов — в `active`.
4. Изменения в коде → сначала обновить документацию.

## 5. Правила для ИИ-ассистента

> Этот раздел определяет поведение ИИ при работе с vault.

### 5.1. Создание новых документов

При создании нового модуля или решения ИИ **обязан**:

1. Использовать подходящий шаблон из `070-Шаблоны/` (Модуль, Тех-Паспорт, UI-Компонент или [[Tpl-Ошибка|070-Шаблоны/Tpl-Ошибка]]).
2. Разместить файл в правильной подпапке `030-Модули/##-Отдел/` или `010-Стандарты/UI-UX/`.
3. Заполнить frontmatter: `title`, `date`, `status`, `tags`.
4. Указать связи через `[[wikilinks]]`.
5. Указать путь к файлам кода в разделе «Техническая реализация».

## 🔡 Работа с Глоссарием

1. **Единый Нейминг**: Если в коде добавляется новое поле (Drizzle) или `pgEnum` — его **обязательно** нужно добавить в `[[010-Стандарты/Глоссарий]]`.
2. **Консистентность**: ИИ обязан проверять глоссарий перед генерацией кода, чтобы не называть `order_category` словом `order_type`.

## 🐛 Фиксация Ошибок

1. **Решил — Запиши**: Если потрачено более 15 минут на поиск бага или причина была неочевидна, создай запись в `[[100-База-Знаний/Энциклопедия-Ошибок]]`.
2. **Анализ**: Заполнение раздела "🛡 Как избежать в будущем" является обязательным для предотвращения повторных инцидентов.

### 5.2. Обновление графа

При добавлении **новой корневой папки** (`###-Название`):
1. Добавить запись в `.obsidian/graph.json` → `colorGroups[]`.
2. Формат: `{ "query": "path:###-Название", "color": { "a": 1, "rgb": <число> } }`.
3. Обновить таблицу в разделе 1 этого документа.

При добавлении **новой подпапки модулей** (`##-Отдел`):
1. Добавить запись в `graph.json` → `colorGroups[]` с `path:030-Модули/##-Название`.
2. Обновить таблицу подгрупп в разделе 1.

### 5.3. Обновление Canvas

При изменении структуры модулей → обновить `Система.canvas`:
- Группы соответствуют подпапкам `030-Модули/`.
- Узлы ссылаются на файлы через `"file": "030-Модули/##-Отдел/Файл.md"`.
- Связи (`edges`) отражают взаимодействия между модулями.

### 5.4. Решения (040-Решения)

ИИ создаёт документ-решение когда:
- Реализация **нетривиальная** (не очевидная из кода).
- Была **решена проблема/баг** с неочевидным фиксом.
- Принято **архитектурное решение** с trade-offs.

Структура: Проблема → Решение → Ключевые особенности → Файлы.

### 5.5. Синхронизация с кодом

При создании нового модуля в `app/`:
1. Создать документ в `030-Модули/` сразу.
2. При удалении модуля — пометить документ `status: deprecated`.
3. Обновлять `progress` и `updated` при существенных изменениях.

## 7. Инструментарий и плагины

### 7.1. Визуальный слой
- **Excalidraw**: Используется для архитектурных набросков, UI-вайрфреймов и сложных схем. 
  - Файлы хранятся в папке соответствующего модуля или в `020-Архитектура/Рисунки/`.
  - Внедряются в заметки через `![[Файл.excalidraw]]`.
- **Mermaid Tools**: Для генерации ER-диаграмм, Sequence-диаграмм и простых блок-схем прямо в тексте.
  - Код диаграммы должен быть обёрнут в ```mermaid.

### 7.2. Управление задачами
- **Tasks**: Все задачи в vault должны следовать синтаксису `- [ ] Название задачи 📅 ГГГГ-ММ-ДД`.
  - ИИ обязан добавлять даты и приоритеты (⏫, 🔼, 🔽) при планировании фич.
- **Kanban**: Глобальный роадмап и состояние модулей отслеживаются на досках в `060-Roadmap/`.
- **Calendar**: Используется для навигации по журналу сессий (`050-Сессии`) и отслеживания дедлайнов задач.

### 7.3. Качество и структура
- **Linter**: Автоматически форматирует заметки при сохранении.
  - Настройки: соблюдение Title-Case для заголовков, удаление лишних пустых строк, стандартизация frontmatter.
- **Colored Tags Wrangler**: Цвета тегов определяют приоритет и статус:
  - `#статус/план` — синий
  - `#статус/активный` — зеленый
  - `#приоритет/P0` — красный
- **DataCards**: (Требует Dataview) Используется для генерации каталогов модулей, агентов и решений в виде карточек.
- **Templater**: Ядро автоматизации. Все шаблоны используют синтаксис `<% tp... %>` для динамической подстановки дат, папок и имен файлов.
- **Git**: Обеспечивает версионность базы знаний и автоматические бэкапы. 
  - Настройки: Рекомендуется включить "Auto Backup" каждые 30-60 минут и "Pull on startup".
