# Стандарты разработки MerchCRM

Этот документ является техническим регламентом для всей кодовой базы MerchCRM.

## 0. Технологический стек
- **Framework**: Next.js 15.5.15 (App Router)
- **Styling**: Tailwind CSS v4
- **Database**: PostgreSQL (Drizzle ORM) — см. [[База-данных]]
- **Security**: Better Auth (Redis)

## 1. Архитектура
- **Domain Services**: Вся бизнес-логика выносится в `lib/services/`.
- **Server Actions**: Только тонкая обертка с валидацией (`zod`).
- **Стриминг**: Использование потоков (`ReadableStream`) для файлов.

## 2. TypeScript (Строгая Типизация)
- **Any-Ban**: Использование `any` категорически запрещено. Запрещено использовать `: any`, приведение `as any`, generic `<any>`. Используйте `unknown` или явную типизацию.
- **Strict Types**: Локальные глобальные интерфейсы (например, объявление `interface Order` или `interface User` прямо в файле компонента) **запрещены**. Все общие типы должны импортироваться из `@/lib/types`.
- **Игноры**: Использование `@ts-ignore` запрещено.

## 3. Server Actions и Безопасность
Все серверные экшены (Server Actions) проходят строгий автоматизированный аудит на уязвимости (особенно IDOR):
- **Инициализация**: Файлы с экшенами обязаны иметь директиву `"use server"`.
- **Авторизация (Auth & RBAC)**: Любой Server Action (кроме публичных) **обязан** проверять сессию: `const session = await getSession(); if (!session) ...`. 
- **Предотвращение IDOR**: При редактировании или удалении сущностей по ID (например, `deletePromocode()`, `updateKBPage()`), функция **обязана** проверять принадлежность объекта пользователю или наличие прав `Администратор` через `requireAdmin()` или проверку БД, прежде чем выполнять мутацию.
- **Обертки**: Используйте `withAuth` или блок `try-catch` для всех мутаций.
- **Финансы (Audit & Logging)**: Любые экшены, касающиеся финансов (зарплаты, P&L, транзакции, фонды), обязаны иметь детальное логирование через `audit()`.
- **Документация**: Обязательно указывать JSDoc-теги `@requires` (роли доступа) и `@audit` (категория лога) для мутаций.

## 3.1. Ошибки рендеринга и Client Components
- **Hydration Mismatch**: Строго запрещено вызывать генераторы случайных данных (например, `Math.random()`) или `new Date()` напрямую в рендере `Client Components`. Выноси эти вызовы в `useEffect` или прокидывай с сервера, иначе происходит конфликт гидратации.
- **Null Safety**: При вызове методов массивов (`.map`, `.filter`, `.length`), переменные вроде `items`, `data`, `orders` должны быть защищены (`items?.map(...)` или `(items || []).map(...)`).

## 4. Рабочие процессы
- **Тесты**: `Vitest` (unit/int) и `Playwright` (e2e).
- **WebP**: Авто-конвертация всех медиа-файлов.
- **Логи**: Все действия в `audit_logs`.

## 5. Актуальные регламенты (Wave 10)
С целью оптимизации производительности и обеспечения целостности данных внедрены следующие правила:
- **Модульная Схема**: Запрещено использование `import * from "@/lib/schema"`. Импорты таблиц должны быть гранулярными (из конкретных файлов модуля). Подробнее: [[Actions]].
- **Ручной updatedAt**: В связи с отсутствием БД-триггеров, каждый `db.update` **обязательно** должен включать `updatedAt: new Date()`. Подробнее: [[База-данных]].

## 6. Тестирование и Моки
- **Реалистичные моки**: Мок-объекты (например, `RedisMock`) **документально обязаны** имитировать поведение реальных сервисов в части пагинации и курсоров.
  - Использовать `RedisMock.scan` с поддержкой курсоров для проверки `do...while` циклов.
  - Моки не должны возвращать все данные сразу, чтобы тесты могли выявлять ошибки неполной обработки батчей.

## 7. Архитектура брендинга (BrandingSettings)
- **Единый источник:** `lib/branding.ts` → `getBrandingSettings()`. Это единственное место загрузки глобальных настроек брендинга из таблицы `systemSettings`.
- **Провайдер:** `components/branding-provider.tsx` → `BrandingProvider` / `useBranding()`. Передает настройки через React Context, устанавливает CSS-переменные (`--primary`, `--radius-outer`, `--radius-inner`).
- **Safe-fail:** При недоступности БД функция возвращает `defaultBranding` — приложение продолжает работать на дефолтной теме без краша.
- **Обязательные поля:** `primaryColor`, `color` (алиас `primaryColor`), `companyName`, `currencySymbol`.
- **Запрещено:** Создавать альтернативные функции загрузки настроек. Реэкспорт допустим: `export const getBrandingSettings = getBrandingSettingsLib;`
