# 🛡️ Безопасность MerchCRM

Основные принципы и технические регламенты обеспечения безопасности системы.

## 1. Защита Server Actions
Все мутации данных через Server Actions **обязаны** использовать многоуровневую защиту:
- **Валидация**: Использование `zod` схем для строгой типизации входящих данных.
- **CSRF**: Обязательное использование `withCsrf` для всех действий, изменяющих состояние БД.
- **RBAC**: Проверка ролей через middleware или системные промпты (для AI).

```typescript
// Пример защищенного экшена
export const updateOrder = createSafeAction(OrderSchema, async (data) => {
  return await withAuth(async (session) => {
    // Внутренняя проверка прав и CSRF токена
    return await orderService.update(data, session.user.id);
  });
});
```

## 2. Аутентификация (Better Auth)

### 2.1. Логирование
- **ЗАПРЕЩЕНО** выводить хеши паролей, токены или секреты в `console.log`.
- Даже частичный вывод bcrypt-хеша раскрывает версию алгоритма, cost factor и начало соли.
- В production `removeConsole: true` (Next.js) не гарантирует удаление из системных логов Docker.

### 2.2. Секреты
- Для шифрования чувствительных данных используется `lib/crypto.ts`.
- Секрет шифрования базируется на `ENCRYPTION_KEY` или `AUTH_SECRET` (fallback).
- **ЗАПРЕЩЕНО** извлекать секреты напрямую из `process.env`. Используйте валидированные значения из `lib/env.ts`.
- Минимальная длина секрета: **32 символа** (валидация через `z.string().min(32)`).

### 2.3. Хеширование паролей
- Алгоритм: **bcrypt** с cost factor **12** (файл `lib/password.ts`).
- Cost factor 10 считается устаревшим для 2026 года.
- Обратная совместимость: bcrypt автоматически верифицирует старые хеши с cost 10.

### 2.4. Доверие прокси-заголовкам
- Параметр `trustedProxyHeaders` управляется через переменную `TRUST_PROXY`.
- **Production**: по умолчанию `false`. Включать **только** при наличии корректно настроенного обратного прокси (Nginx).
- **Development**: всегда `true` для удобства локальной разработки.
- Без ограничения атакующий может подделать IP через `X-Forwarded-For`.

## 3. Работа с БД (Drizzle ORM)
- **SQLi**: СТРОГО запрещено использование конкатенации строк. Только `sql` теги Drizzle.
- **Транзакции**: Все связанные изменения (заказ + склад + финансы) должны быть атомарными.

## 4. Инфраструктура Docker

### 4.1. PostgreSQL
- Порт `5432` **НЕ пробрасывается** на хост в production (`docker-compose.prod.yml`).
- Доступ к БД — только через внутреннюю сеть Docker (`app_network`).
- Для прямого доступа в production использовать `docker exec` или SSH-туннель.
- **Дефолтные пароли запрещены** — переменная `POSTGRES_PASSWORD` обязательна (без fallback).

### 4.2. Redis
- Healthcheck использует аутентификацию: `redis-cli -a ${REDIS_PASSWORD} ping`.
- Данные персистятся через том `redis_data:/data` (сессии сохраняются при рестарте).
- Доступ защищен паролем через `--requirepass`.

### 4.3. Nginx Proxy Manager
- Обязан очищать и перезаписывать заголовок `X-Forwarded-For` реальным IP клиента.
- Управление SSL: порт 81 (внутренний).

## 5. Переменные окружения

> [!WARNING]
> Все переменные окружения **обязаны** проходить через Zod-валидацию в `lib/env.ts`. Прямой доступ к `process.env` **запрещён** в бизнес-логике.

| Переменная | Назначение | Обязательность |
|---|---|---|
| `AUTH_SECRET` | Секрет аутентификации и шифрования (fallback) | ✅ Обязательна |
| `ENCRYPTION_KEY` | Основной ключ AES-256 (32 байта) | ⚙️ Желательна |
| `POSTGRES_PASSWORD` | Пароль PostgreSQL | ✅ Обязательна |
| `REDIS_PASSWORD` | Пароль Redis | ✅ Обязательна |
| `TRUST_PROXY` | Доверять прокси-заголовкам | ⚙️ По умолчанию `false` |

## 6. Файловое хранилище
- **Image-Src**: Поддержка S3 хранилища (Reg.ru) и локального `local-storage/`.
- **WebP**: Автоматическая конвертация при загрузке для снижения риска внедрения вредоносных данных в метаданные.

## 7. CSP & Принудительная изоляция (Middleware)
В проекте используется Content Security Policy (CSP), управляемая через `middleware.ts`.

- **Режим**: Исполнение на основе `nonce`. Все скрипты и стили должны иметь валидный nonce.
- **Исключения (API)**: Маршруты `/api/*` **исключены** из CSP. 
    - Это предотвращает лишнюю генерацию nonce для JSON-ответов и снижает нагрузку на сервер.
    - Security-заголовки (HSTS, X-Frame-Options) для API сохраняются.
- **Исключения (Calculators)**: Для маршрута `/dashboard/production/calculators` разрешен `unsafe-eval`. 
    - Это необходимо для работы Fabric.js и Three.js. 
- **Хардкенинг (Style-Src)**:
    - Используется стратегия **Nonce + Fallback**: `'nonce-${nonce}' 'unsafe-inline'`.
    - `'unsafe-inline'` сохраняется как fallback для старых браузеров и для поддержки атрибутов `style={{...}}` в React (например, в Recharts, Framer Motion), которые не поддерживают nonce.
- **Защита Nonce**:
    - Запрещено передавать nonce в заголовках ответа (`x-nonce`). Nonce должен быть доступен только внутри Server Components через `headers()` и внедряться в DOM.
- **Хардкенинг (Crypto)**:
    - Функция `safeCompare` защищена от **Length Oracle** атак через предварительное хеширование SHA-256. Это предотвращает утечку длины токена через время ответа.
