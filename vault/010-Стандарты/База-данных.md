# Стандарты работы с базой данных

Этот документ определяет правила взаимодействия с PostgreSQL через Drizzle ORM в проекте MerchCRM.

## 1. Индексация
Для обеспечения высокой производительности запросов индексы **обязательны** для:
- Всех внешних ключей (`xxx_id`).
- Полей статусов (`status`, `type`).
- Полей логического удаления (`is_archived`, `deleted_at`).
- Полей дат частого поиска (`created_at`).

## 2. Транзакции
Любое изменение данных, затрагивающее более одной таблицы или требующее целостности нескольких операций, **должно** выполняться внутри блока `db.transaction`.

```typescript
await db.transaction(async (tx) => {
  await tx.insert(orders).values(orderData);
  await tx.update(inventory).set({ quantity: ... });
});
```

## 3. Атомарность
Для обновления числовых счетчиков (количество на складе, баланс) используйте SQL-выражения вместо предварительного чтения значения. Это предотвращает состояние гонки (race conditions).

```typescript
// ПРАВИЛЬНО
await db.update(inventoryItems)
  .set({ quantity: sql`${inventoryItems.quantity} - 1` })
  .where(eq(inventoryItems.id, itemId));
```

## 4. Предотвращение N+1
Строго запрещено выполнять SQL-запросы внутри циклов. 
- Используйте `db.query.findMany` с параметром `with: { ... }` для предварительной загрузки (eager loading) связанных данных.
- Для сложных случаев используйте `JOIN` или `json_agg`.

## 5. Миграции
Все изменения схемы **обязаны** проходить через `drizzle-kit generate` и фиксироваться в папке `drizzle/`. 

- **Local**: `npm run db:generate` для создания SQL-файла.
- **Verification**: Проверка миграций в PR через GitHub CI.
- **Production**: Автоматическое применение через сервис `merch-crm-migrator` (скрипт `scripts/db-migrate.ts`) при деплое.
- **Strict**: Прямое редактирование схемы на проде и использование `drizzle-kit push --force` запрещено для защиты данных.

## 6. Автоматическое обновление (updatedAt)
В проекте MerchCRM настроены PostgreSQL триггеры для автоматического обновления поля `updated_at`. 

- **Триггер**: `trg_set_updated_at_<table_name>`
- **Функция**: `set_updated_at()`
- **Действие**: При выполнении `UPDATE` поле `updated_at` автоматически принимает значение `NOW()`.

Разработчикам **не требуется** вручную передавать `updatedAt: new Date()` в методах `.set()` при использовании Drizzle ORM для таблиц, где этот триггер активирован.

## 7. Синхронизация статистики (Client Stats & RFM)
Для обеспечения целостности денормализованных данных в таблице `clients` используется триггер уровня БД:

- **Триггер**: `trg_sync_client_stats` на таблице `orders`.
- **Функция**: `update_client_stats_from_orders()`.
- **Логика**: При любом изменении заказов автоматичски пересчитываются:
    - `totalOrdersCount`, `totalOrdersAmount`, `averageCheck`.
    - Даты первого и последнего заказов.
    - **RFM-метрики**: Recency (давность), Frequency (частота), Monetary (деньги).
    - **RFM-сегмент**: Автоматическое определение сегмента (Champions, Loyal, At Risk и т.д.).

Это гарантирует актуальность аналитики даже при прямых манипуляциях с базой данных через SQL.
