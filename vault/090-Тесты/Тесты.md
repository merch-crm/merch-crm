## 🛠 Технический стек
- **Runtime**: Vitest 2.1.9.
- **E2E**: Playwright (запланирован).
- **CI/CD**: GitHub Actions (автозапуск при Push и Pull Request).
- **Redis мок**: `RedisMock` — in-memory Map, не требует реального Redis.

## 🧩 Архитектура тестов

### Утилиты (`test-utils/db.ts`)
- `getTestDb()` — пул соединений (max: 10, timeout: 30s).
- `cleanupTestDb()` — единый `TRUNCATE ... CASCADE` по всем таблицам.
- Файловое логирование в `/tmp/vitest-db.log` для диагностики.

### Мокирование
- **Redis**: `RedisMock` в `lib/redis-mock.ts`.
  - **SCAN & Cursors**: Поддерживает реалистичную пагинацию через курсоры (на базе имен ключей). Это позволяет тестировать сложные циклы `do...while`, имитируя поведение реального Redis.
  - **Безопасность**: Мок корректно обрабатывает удаление ключей во время сканирования.
- **Queue tasks**: `vi.mock()` для `queueClientStatsUpdate()` и подобных.
- **Auth**: `vi.mock()` для `getSession()`.

### Конфигурация (`vitest.config.ts`)
```typescript
test: {
  testTimeout: 60_000,
  hookTimeout: 60_000,
  setupFiles: ['./test-utils/setup.ts'],
}
```

## ⚠️ Известные особенности
1. **SSH-зависимость**: Интеграционные тесты подключаются к удалённой БД через SSH-туннель. При нестабильном соединении возможны таймауты.
2. **Рекомендация**: Рассмотреть переход на локальный Docker PostgreSQL для изолированного запуска тестов.
