# Кеширование через Redis

## Проблема
Next.js в production запускает несколько воркеров. Стандартный `Map`-кеш не работает — каждый воркер имеет своё состояние. При высокой нагрузке одни и те же SQL-запросы (статистика заказов, список клиентов) выполняются повторно.

## Решение
Класс `RedisCache` (`lib/cache/redis-cache.ts`) — централизованный кеш, доступный всем воркерам.

### Архитектура
```
┌──────────┐    ┌──────────┐    ┌──────────┐
│ Worker 1 │    │ Worker 2 │    │ Worker 3 │
└────┬─────┘    └────┬─────┘    └────┬─────┘
     │               │               │
     └───────┬───────┘───────┬───────┘
             │               │
        ┌────▼───────────────▼────┐
        │     Redis (ioredis)      │
        └──────────────────────────┘
```

### Ключевые паттерны

**1. getOrSet (Cache-Aside)**
```typescript
const result = await redisCache.getOrSet(
    "crm:stats:orders:30d",
    async () => db.query.orders.findMany(...),
    { ttl: 60 }
);
```
Если ключ есть в кеше → возвращает из Redis. Если нет → выполняет запрос, кладёт в кеш, возвращает.

**2. Стандартизированные ключи и TTL**
```typescript
CACHE_KEYS.orderStats("30d")  → "crm:stats:orders:30d"
CACHE_TTL.ORDER_STATS          → 60 секунд
CACHE_TTL.CLIENTS_LIST         → 300 секунд
CACHE_TTL.BRANDING             → 900 секунд
```

**3. Паттерновая инвалидация**
```typescript
await redisCache.invalidatePattern("crm:stats:orders:*");
// Удаляет ВСЕ ключи статистики заказов
```

### Устойчивость к сбоям
- `lazyConnect: true` — подключение только при первом запросе
- `reconnectOnError` — автоматическое переподключение при READONLY / ECONNRESET
- `maxRetriesPerRequest: 3` — ограничение повторных попыток
- При недоступности Redis приложение работает без кеша (graceful degradation)
