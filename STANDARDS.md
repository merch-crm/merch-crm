# Стандарты Разработки и Дизайна MerchCRM

Этот документ описывает ключевые правила, стандарты разработки и гайдлайны по UI/UX дизайну, утвержденные для проекта MerchCRM.

## 🛠 Стандарты Разработки (Development Standards)

### 1. Архитектура и Фреймворк
- **Стек:** Next.js (App Router), React 18, TypeScript, Tailwind CSS, Drizzle ORM.
- **Серверные экшены (Server Actions):** Вся бизнес-логика и взаимодействие с БД (Drizzle ORM) инкапсулируются в Server Actions (файлы `*-actions.ts`). 
- **Клиентские компоненты:** Используются только там, где нужна интерактивность (`"use client"`). Максимальное использование Server Components (`RSC`) для оптимизации производительности.

### 2. Строгая типизация (TypeScript)
- **Запрет `any`:** Использование типа `any` категорически запрещено. Все переменные, параметры функций и возвращаемые значения должны иметь строгие типы.
- **Исключения:** Разрешено использование `// @ts-expect-error` только при интеграции с библиотеками без типов, с обязательным комментарием, объясняющим причину.
- **Валидация:** Все входные данные (формы, параметры API, аргументы Server Actions) должны валидироваться через **Zod**.

### 3. Обработка ошибок
- **Стандартизированный возврат:** Server Actions всегда возвращают объект типа `DashboardActionResult<T>` (или аналогичный), например: `{ success: true, data: T } | { success: false, error: string }`.
- **Try/Catch:** Ошибки необходимо перехватывать глобальными `try/catch`.
- **Логирование:** Для регистрации системных ошибок используется утилита `logError()` из `@/lib/error-logger`. Никаких `console.log` или `console.error` в Production-коде.

### 4. Безопасность (Security)
- **Авторизация:** Каждое защищенное действие (Server Action, API Route) должно начинаться с проверки сессии: `const session = await getSession(); if (!session) return { ... }`.
- **RBAC (Управление ролями):** Чувствительные действия (удаление, экспорт) проверяются по ролям: `if (!["Администратор", "Руководство"].includes(session.roleName)) { ... }`.
- **Аудит критических действий:** Любое изменение, удаление или скачивание важных данных (пользователи, настройки, бэкапы) логируется в БД через `logAction("Событие", "entity", id)`.

## 🎨 Стандарты UI Дизайна (UI/UX Standards)

Дизайн должен быть премиальным, консистентным и интуитивно понятным, следуя философии "Vibrant Lime" и современным трендам (стекломорфизм, чистые интерфейсы).

### 1. Цветовая палитра и Иерархия фонов
- **Основной фон приложения:** Светло-серый (`bg-slate-50` / `bg-gray-50`) для визуального отделения контентных карточек.
- **Карточки и Контейнеры:** Белый цвет (`bg-white`) с мягкими тенями (`shadow-sm`, `shadow-md`) и скругленными углами (`rounded-xl` или `rounded-2xl`).
- **Модальные окна, Шторки (Drawers) и Popover:**
  - **Фон главного контейнера (контента):** Всегда чистый белый (`bg-white`).
- **Поля ввода и интерактивные блоки (Внутри карточек и модалок):**
  - **Поля ввода (Inputs, Textareas):** В состоянии покоя светло-серые (`bg-slate-50`). 
  - **Фокус:** При фокусе или активном редактировании поле переходит на белый фон (`focus:bg-white`) для выделения акцента. Обязательна обводка фокуса в фирменном цвете (`focus:ring-2 focus:ring-primary/50`).
  - **Списки и Select'ы:** Аналогично полям ввода — серый фон в закрытом виде, белый в открытом или в фокусе.

### 2. Типографика и Текст
- Основной шрифт: **Inter** или аналогичный современный гротеск (санс-сериф).
- **Заголовки:** Четкая иерархия. `h1` для названий дашбордов (`text-2xl font-bold`), `h2`/`h3` для карточек.
- **Цвета текста:** `text-slate-900` для основного текста, `text-slate-500` для второстепенного (подписи, описания).

### 3. Иконки и Индикация
- Иконки (Lucide React) используются только там, где они несут функциональный смысл или значительно ускоряют считывание списка.
- Декоративные и бессмысленные иконки (например, замок рядом с очевидными настройками профиля) запрещены во избежание визуального шума.

### 4. Анимации (Micro-interactions)
- Обязательно использование утилит транзиции: `transition-all duration-200`.
- При наведении на кликабельные элементы (кнопки, строки таблиц) должен меняться фон (`hover:bg-slate-50` для белых карточек) или тень.
- Использование плавных появлений для модальных окон (Fade-In, Slide-Up).

### 5. Структура Dashboards (Bento Grid)
- Использование "Bento Grid" (модульная сетка карточек) для аналитики и метрик.
- Карточки должны быть выровнены (стабильная высота/ширина) и адаптивны для мобильных устройств.
